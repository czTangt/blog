记录一下 vscode 及其相关插件的配置和使用。

本体操作

工作区，目录和 Profiles

对于目录而言，我们可以通过设置 settings.json 文件来更改相关配置。这里我们打开 Settings，使用的不是 User 的全局配置，而是使用 Workspace 的工作区配置，这样我们就会在当前目录的根目录生成一个 .vscode 目录，里面就存放着我们当前目录的 setting.json 文件，它就是对于当前目录的配置。也就是说，这个时候单独的一个目录就是一个工作区。

而对于工作区，我们就是设置了一个相关配置文件，里面存放着我们子项目的地址和此工作区的 settings.json 文件，同时每个子项目下面还存在着 .vscode 目录。在这种情况下，我们可以设置工作区配置，也可以对单个子项目进行配置。它这种配置方案更为细致。但是缺点就是每次打开相关项目，要想按照之前的配置，那么必须通过打开工作区的方式来打开，也就是一次性打开同一个工作区下面的数个子项目。并且工作区配置文件的路径为项目地址，原项目移动地址之后就不在这个工作区内了。因此我感觉它不适宜我现在项目随机分配的情况。

对于 Profiles 而言，它就是新设置了一个配置方案，可以自己设置哪个目录采用哪种配置。但是根据实际尝试，它的插件都是单独的，也就是每一个配置，都要重新下载一遍相关插件，这个是完全不能忍受的，所以也放弃了这种方案。

最后还是把每个目录当作一个工作区，然后在这里进行插件的禁用。虽然说目前大部分语言相关的插件都是自动加载，也就是只有用到的时候才会进行加载，在加载之后会在插件的右上角显示加载时间。但是据我观察，有些插件感觉没有进行使用就加载了，所以面对不同的目录，还是采用禁用不相关插件的办法。

插件相关

PicGo

上传配置

因为使用 vscode 来写 Markdown 语法，所以想着仿照之前 Typora + PicGo 配置的剪切板粘贴上传图片功能。同时因为之前在 PicGo 应用中配置过，所以这次就直接获取参数然后输入即可。

这里的配置信息直接从软件中抄过来，或者随便看看阿里云图床的配置文章就可以知道大致该怎么填写了。之后这里 Aliyun: Path 选择之前自己创建的目录，可以通过 OSS -> Bucket -> 文件列表 来查看。另外选择 Pic Bed: Current 为 aliyun，这样上面的配置才可以生效。同时还设置了快捷键，对于使用截图软件，把图片放在剪切板上的情况，修改快捷键为 ctrl + alt +w，这样和普通粘贴分割开了，更加方便。

文件名设置

Custom Output Format 就是文件在 vscode 中的展现形式，这里针对 Fixit 主题中 Image 的要求 进行了修改。这样后面的 title 可以直接生成，然后我们需要修改的就是前面的 alt 了。 layouts/partials/plugin/image.html 文件在之前进行修改了，所以图片下面的图标显示的就只有 alt 了，只有鼠标放在图片上面和放大图片显示的才是 title。

Custom Upload Format就是上传文件的格式，随便设置即可，因为不会有太多的格式要求，这里就是默认的配置。

保存路径

这里有一个 Data Path，vscode 插件会把每次的提交形成 json 数据添加到这个文件中，所以可以设置存储路径，之后查看图片可以迅速找到相关信息。这里我也将路径设置为博客的仓库，使其作为附属信息上传到 github 上。

java 环境配置

下载

主要使用 wsl 来配置 java 环境来做实验，所以对于 java 的下载直接使用包安装工具来进行。 配置环境主要需要两个插件 Language Support for Java(TM) by Red Hat 和 Extension Pack for Java，然后它们会连续下载多个插件，一般这两个下载完了就配置了基础环境。

项目创建

之后我们就可以在 Ctrl + P 打开的配置栏中创建 java 项目了。

jar 包添加

如果我们想要添加外部下载的 jar 包，我们可以使用 Maven 来进行配置（之前插件也一起下载了）。但是我这里没有相关环境，所以采用手动导入的方式。因此可以在一个 java 项目中打开 java 文件，可以看见下面的 JAVA PROJECTS 标识，它可以进行相关配置。 我们可以直接点击 Referenced Libraries 右侧的 + 直接在文件系统中找自己下载的 jar 包。

或者也可以直接如下图所示在配置界面进行添加。

jupyter

配置环境

我是用的就是 miniconda 环境。下载并配置环境之后，在 vscode 中下载插件 python 和 jupyter。之后在 settings 中输入 python path ，然后输入 miniconda -> envs -> python.exe 的文件位置，这样之后就会直接使用默认的 python 解释器（我猜的，还没有经过验证）。

介绍

对我来说，Jupyter Notebook 在编程时具有语法高亮、缩进、tab补全的功能。并且可以在代码块下方展示运行结果。对代码编写说明文档或语句时，支持 Markdown 语法，支持使用 LaTeX 编写数学性说明。我主要想的就是可以通过划分不同的代码块来加强代码的理解，同时还可以使用 markdown 语法来进行讲解，这个比 python 需要最后再运行强很多，我需要的就是这种划分代码段分别运行，最后还能关联在一起的效果。这里之后根据 官方指南 进行 jupyter 的讲解。

运行细胞

在这里，每一个代码块被称为一个 cell。要运行代码，可以在命令和编辑模式下使用键盘快捷键。要运行当前单元格，使用 Ctrl+Enter。要运行当前单元格并前进到下一个单元格，使用 Shift+Enter。之后也可以使用图形界面的选项来进行相关操作。

上文提到了命令模式和编辑模式，所以这里需要知道运行细胞存在三种个状态：未选中、命令模式和编辑模式。代码单元格和编辑器边框左侧的垂直条显示单元格的当前状态。

  • 当没有可见的栏时，该单元格未被选中。选择单元格后，它可以处于命令模式或编辑模式
  • 在命令模式下，单元格左侧将出现一个实心垂直条，该单元可以进行操作并接受键盘命令
  • 在编辑模式下，单元格编辑器周围有一个实心垂直条由边框连接起来，单元格的内容可以修改
  • 在键盘上，按 Enter 键可进入编辑模式，按 Esc 键可进入命令模式。

注意

有的主题不能呈现上述的颜色条变化，所以为了兼顾主题和 jupyter，我选择 Github Theme 中的 Complete Dark 主题，起码比较而言，它可以兼顾两者。后面发现这个主题是在别的插件共同渲染下才满足我的想法的，把别的插件禁用之后就不太好看了，所以又找了一个插件 Learn with Sumit Theme，选择 Blue Velvet 主题。
